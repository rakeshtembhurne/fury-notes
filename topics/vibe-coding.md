# Vibe Coding

## What Is Vibe Coding?

**Vibe coding** is a term coined to describe building software with AI agents where you provide high-level direction rather than line-by-line code. You "vibe" the outcome you want, and the AI handles implementation.

---

## The Philosophy

### Naval's Core Argument

**"AI coding agents can now deliver one-shot custom apps straight from a tweet."**

**What Changed:**
| Before | Now |
|--------|-----|
| Multiple prompts refining spec | Single prompt → working app |
| Back-and-forth debugging | AI handles edge cases first time |
| Human intervention on failures | Minimal human skill needed to direct |
| Skill gatekeeping required | Anyone with vision can build |

### Why This Matters

1. **Builders emerge without coding skill** - Idea execution speed > coding ability
2. **Founder advantage shifts** - Who can articulate vision, not who can code
3. **Velocity explosion** - Days/weeks → hours
4. **Gatekeeping dissolves** - Coding knowledge no longer a moat

---

## Vibe Coding vs Traditional

| Aspect | Traditional Coding | Vibe Coding |
|--------|-------------------|-------------|
| **Control** | Explicit, line-by-line | Directional, high-level |
| **Debugging** | Manual, deep | Light touch, AI handles |
| **Iteration** | Full rewrites | Small adjustments |
| **Skill required** | High (syntax, logic) | Low (vision, direction) |
| **Speed** | Slow (weeks/months) | Fast (hours/days) |
| **Gatekeeping** | Coding knowledge | None |

## How "Vibing" Works

### The Vibing Process

1. **Trust-based:** You trust the AI output without verification
2. **Directional, not prescriptive:** "Build me X" not "here's exact spec"
3. **Iterative but lightweight:** Slight redirects, not rewrites
4. **Flow state:** Builder stays in ideation, not debugging
5. **Emergent solutions:** AI finds approaches humans wouldn't write

## Vibe Planning

A smarter approach for new developers:

1. **Brainstorm** with a coding agent (different from implementation agent)
2. **Ask for feedback** - Request clarifying questions
3. **Divide and conquer** - Plan atomic parts separately
4. **Pick one feature** at a time
5. **Implement** with separate coding agent
6. **Test & verify** using initial agent
7. **Iterate** until satisfied

---

## Spec-Driven Development

The problem with vibe coding: **lack of specifications**

When you give simple text prompt, coding agents must make assumptions → errors.

**The Solution - 4 Steps:**

1. **Specify** - What and why (not tech stack)
2. **Plan** - Technical details, architecture, tech stack
3. **Tasks** - Manageable, actionable tasks
4. **Execute** - Parallel or sequential

### GitHub Spec Kit

GitHub's implementation of spec-driven development:
- Works with existing coding agents (Cursor, Claude Code, etc.)
- 4-step process
- Cheaper than AWS Kirr

## Problems with Vibe Coding

### The 70% Problem

> "The world of vibe coding is littered with 70% complete projects that have zero users."

**Why:**
- Building ≠ selling
- Marketing is often seen as "lesser than"
- No users = no feedback loop

### Solutions

1. **Marketing-first approach** - Define success metrics before building
2. **Get users early** - Test ideas before full implementation
3. **Structured workflows** - Don't skip software engineering process
